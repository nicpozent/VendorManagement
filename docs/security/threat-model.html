Vendor Technical Review — Threat Model & Security Analysis

A STRIDE + LINDDUN assessment of the "Assess" platform: a React 18 SPA and a .NET 8 API on PostgreSQL, secured with Microsoft Entra ID and integrated with Microsoft Graph for administrator user-import and email reminders. The review is drawn from the code as built — the auth handlers, the authorization checks, the Graph client, the reminder worker and the data model — and states gaps rather than smoothing them. The core (identity, server-authoritative authorization, verdict integrity, injection resistance) is sound; the open items are pre-production hardening — email-abuse containment, secrets, at-rest privacy, auditability and CI security gates. Update: two findings raised here have since been fixed in code and runtime-verified — the dev-auth fallback now fails closed outside Development, and NDA/reminder mail is rate-limited and recipient-allowlisted; the tables below mark these Fixed.

System & trust boundaries

Five trust boundaries: browser → edge, edge → API, API → PostgreSQL, API → Entra ID, and API → Microsoft Graph. The API is the sole authority — the SPA is untrusted, its role switch is a display hint, and verdict/authorization are decided server-side.

Verdict

Moderate (3.4/5, ↑ from 3.1 after this cycle's fixes) — architecturally sound, safe for internal pilot behind Entra; not yet hardened for production or external exposure

The platform gets the hard parts right. Authentication is delegated to Microsoft Entra ID and the API validates the token on every call; authorization is server-authoritative — the client's role switch is a display hint only, managers are scoped to their own reviews, and admin-only surfaces (entities, directory import, settings) are gated in the handler. The verdict, readiness and completeness are computed on the server, so a client cannot forge an approval, and data access is through EF Core parameterized queries with React's default output-encoding on the memo — the classic injection classes are closed. This cycle closed two of them. The local dev-auth fallback — which authenticated every request as CFO/admin when Entra settings were absent — now fails closed outside Development (verified: the API refuses to start rather than run open). The email actions (send-nda, remind) are now per-user rate-limited and recipient-allowlisted (verified), removing the spam/phishing pivot. The one remaining High is a tenant action, not code: the app-only Mail.Send grant is tenant-wide, so a leaked client secret could send mail as any mailbox — constrain it to one shared mailbox via an Application Access Policy and move the secret to a managed identity or certificate credential in Key Vault. Rounding out the Medium tier: personal data (vendor contacts, imported directory users) is stored in the clear and wants at-rest/column encryption plus a DPIA/ROPA; there is no audit trail for sign-off, verdict changes, imports or NDA sends; and there is no CI security gate (SAST/SCA/secret-scan) yet — one dependency advisory was already patched by hand. Operationally it is single-node with volume-snapshot backup only. None of these block an Entra-gated internal pilot; all should close before production or any external-facing deployment. Frameworks in scope: GDPR (personal data + LINDDUN privacy threats), with ISO 27001 / SOC 2 control mapping recommended as the platform matures.

Basis: code as built (auth handlers · authorization checks · Graph client · reminder worker · EF model) at review time. STRIDE per element + LINDDUN privacy overlay. Honest-assessment basis: gaps are stated, not smoothed over. Not a substitute for an independent penetration test.
